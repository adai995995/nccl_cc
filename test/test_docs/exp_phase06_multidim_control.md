# 阶段 6：多维控制策略（window + pacing + channel）

## 1. 目标

在 **window-only（或当前主 actuator）已在目标场景稳定正收益** 且 **detector 误触与恢复可接受**（阶段 2 达标）的前提下，按阶梯引入 **pacing**、**channel 并发** 等额外自由度，检验是否在 **host+net 混合** 或 **多路径拥塞** 下带来**额外** tail 收益，且副作用可控。

本阶段对应讨论材料 `test3.md` 的工程化落地，**不替代**阶段 0–5 的最小叙事。

## 2. 前置条件（硬门槛）

- 阶段 2：clean 段误触率与恢复延迟在约定阈值内（阈值由项目自行定义并写入 `run_meta.txt`）。
- 阶段 3：默认场景下 Oracle 与 v2-minimal 相对 baseline 的 tail 故事清晰。
- 阶段 4：已明确各单 actuator 的边际效果，避免未消融直接堆叠。

## 3. 阶梯策略（建议顺序）

| 阶梯 | 策略 | 成功判据（建议） |
|------|------|------------------|
| S1 | 收紧 window 路径（detector + window 动态范围） | 不劣于阶段 3 的 v2-minimal 主指标 |
| S2 | S1 + pacing（固定触发规则，如仅 net_stress 或 mixed 标志） | 相对 S1，STRESS 段 p99/max 进一步下降或持平且带宽不降 >X% |
| S3 | S2 + channel 限制（弱触发、窄范围） | 相对 S2，无显著震荡增加；否则回退 S2 |

`X%` 由项目定义（例如 5%），并在报告中固定。

## 4. 自变量 / 因变量 / 控制变量

- **自变量**：组合策略（S1/S2/S3）、pacing/channel 的参数上界、触发门控（连续 K 个 epoch、混合压力标志等）。
- **因变量**：分阶段 tail、吞吐、阶段 2 的 detector 指标（防止多维引入误触爆炸）。
- **控制变量**：与前一阶段相同的 bench 与拓扑；每次只增加一个维度。

## 5. 操作步骤

1. 固定基准为当前最优 **S1** 构建与配置。
2. 引入 **S2**：仅增加 pacing 相关代码路径与 env；跑 **≥5** 次 A/B（S1 vs S2）。
3. 若 S2 稳定，再引入 **S3**；若 S3 震荡或误触上升，记录并默认保留 S2。
4. 混合压力场景：host+net 同时存在时重复上述阶梯（需阶段 3 net 场景就绪）。

## 6. 产物清单

- 每阶梯每 run：完整契约产物 + `policy_id`（如 `S2_pace_v1`）在 `run_meta.txt`。
- `multidim_compare.csv`：列含 policy_id、各阶段 p99/max/skew、带宽、误触率、震荡次数。

## 7. 退出标准

- 至少在一个「混合压力」或「多路径」场景下，**S2 或 S3** 相对 **S1** 有可重复改进（或诚实记录无改进并停止加复杂度）。
- 文档结论与 `test1.md` 一致：**不为复杂而复杂**；无收益则维持 window-first。

## 8. 风险与注意事项

- 多维同时调参易导致不可复现；务必 **单阶梯、少参数**。
- 与论文叙事对齐：多维是 **optional enhancement**，核心命题仍应是「持续局部压力下的 tail 放大与 containment」。

## 9. 仓库对应关系

- 实现位于 NCCL 补丁与 env；实验脚本可复用 `run_timeline_experiment*.sh`，通过 `POLICY=S1|S2|S3` 等 env 切换并在 meta 中记录。
